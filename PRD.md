## 1. Обзор и цели

  

### Что это

  

**Нейрошоу** — платформа-агрегатор форматов интерактивных шоу, в которых несколько LLM-персонажей взаимодействуют по правилам шаблона, а не по написанному сценарию. Драма возникает органически из структуры: правил, стимулов, приватной информации, каналов общения и финальных решений.

  

### Ключевое отличие от "AI-чата"

  

Система не генерирует контент свободно. Она **ограничивает, структурирует и направляет** LLM-вызовы через детерминированный движок-оркестратор. Оркестратор — это не нейросеть; это код с правилами.

  

### Цели продукта

  

- Создать движок, способный запускать любой формат шоу, описанный через шаблон (Show Format Template)

- Обеспечить **Replay** (гарантированный): проигрывание сохранённых raw-ответов из `llm_calls` — точное воспроизведение выпуска без новых LLM-вызовов

- Разделить "ядро" от визуализации и озвучки так, чтобы их можно было подключать независимо

- MVP: запустить один выпуск с 5 персонажами, ведущим-оркестратором и дебаг-интерфейсом в браузере

  

---

  

## 2. Целевая аудитория

  

### Первичная аудитория (MVP)

  

**Разработчик/создатель контента** — человек, который проектирует форматы шоу, создаёт персонажей и запускает выпуски. На этапе MVP это один человек (автор системы), использующий её как инструмент для отладки и экспериментов.

  

### Вторичная аудитория (пост-MVP)

  

- Зрители онлайн-шоу (через публикацию записей выпусков)

- Другие создатели контента, использующие платформу для запуска собственных форматов

## 3. Архитектурная концепция

  

### Ключевые принципы

  

**Разделение ответственности:**

  

```

Show Format Template  →  описывает правила, фазы, каналы, тип финала

Character Definition  →  описывает роль (prompt, параметры, ограничения)

Model Adapter         →  подключает конкретную LLM к роли (сменный "актёр")

Orchestrator Engine   →  детерминированный движок, управляет таймлайном

Event Journal         →  полная история выпуска, основа воспроизведения

Visualization Layer   →  читает журнал, отображает происходящее (отделена от ядра)

```

  

**Информационная изоляция персонажей:**

  

Каждый персонаж получает только то, что он "мог видеть и слышать":

- Свой приватный контекст (карты, цели, секреты) — задаётся до начала

- Публичные сообщения, произнесённые при нём

- Приватные разговоры, в которых он участвовал

  

Персонаж **не имеет доступа** к чужим приватным разговорам. Это не настройка — это архитектурное ограничение на уровне Context Builder.

  

**Детерминированность оркестратора:**

  

Оркестратор управляет выпуском через код, не через LLM. LLM используются только для генерации реплик персонажей в рамках заданных ограничений.

## 4. Основные модули и функциональность

  

### 4.1 Orchestrator Engine (Движок-оркестратор)

  

Центральный детерминированный модуль. Не является LLM.

  

**Функции:**

- Держит таймлайн: фазы → раунды → ходы

- Выбирает, кто ходит следующим, по правилам шаблона (с учётом `speakFrequency` персонажа)

- Определяет канал коммуникации для каждого хода: PUBLIC, PRIVATE(A→B), ZONE(group)

- Выдаёт триггеры по шаблону: вопрос ведущего, событие-условие, обязательное решение

- Следит за лимитами: длина реплики (max tokens), формат ответа (свободный / структурированный / выбор из вариантов)

- Фиксирует каждое событие в Event Journal

- Завершает фазы по условиям (таймер, количество ходов, флаг события)

- Поддерживает режимы управления: AUTO (полный автомат) и DEBUG (пауза/шаг/откат)

- Управляет токен-бюджетом: переходит в `budget_saving_mode` при 80%, запускает `graceful_finish` при 100% (см. раздел 8)

  

**MVP-правило: LLM не инициирует ход**

В MVP оркестратор всегда решает, кто говорит и когда. LLM только отвечает на запрос — она не может самостоятельно взять слово, прервать другого или инициировать приватный разговор без того, чтобы оркестратор назначил ей ход. Это правило упрощает управление темпом и дебаг.

  

Расширение "инициатива персонажа" — Non-MVP, описано в разделе 13.

  

**Acceptance criteria:**

- Оркестратор запускает выпуск от начала до финального решения без участия человека в AUTO-режиме

- В DEBUG-режиме человек может: поставить на паузу в любой момент, выполнить один шаг, откатить фазу к предыдущей точке (на основе журнала)

- **MVP:** оркестратор никогда не передаёт инициативу LLM — LLM только отвечает на назначенный ход, не инициирует его

- В `budget_saving_mode` выпуск остаётся связным и доходит до финального решения без аварийного обрыва

  

### 4.2 Show Format Template (Шаблон формата шоу)

  

Конфигурационный файл (JSON/YAML), описывающий полные правила одного формата шоу.

  

**Поля шаблона:**

```

id: string                        // уникальный идентификатор формата

name: string

description: string

minParticipants: number

maxParticipants: number

  

phases: Phase[]                   // массив фаз в порядке выполнения

  Phase:

    id: string

    name: string

    type: "discussion" | "voting" | "private_talks" | "decision" | "revelation"

    durationMode: "turns" | "timer" | "condition"

    durationValue: number | ConditionExpression

    turnOrder: "sequential" | "frequency_weighted" | "host_controlled"

    allowedChannels: ChannelType[]

    triggerTemplate: TriggerTemplate | null

    completionCondition: ConditionExpression

    // Двойная шкала времени:

    dayIndex: number | null           // если шоу использует "дни" — к какому дню принадлежит фаза

    slotLabel: string | null          // метка слота внутри дня: "утро" | "день" | "вечер" | null

  

days: DayConfig[] | null            // опционально: группировка фаз в "дни"

  DayConfig:

    dayIndex: number

    label: string                     // "День 1", "Финальный день" и т.д.

    phaseIds: string[]                // фазы этого дня в порядке выполнения

  

decisionConfig: DecisionConfig

  DecisionConfig:

    timing: "simultaneous" | "sequential"

    visibility: "secret_until_reveal" | "public_immediately"

    revealMoment: "after_all" | "after_each"

    format: "choice" | "free_text" | "ranking"

    options: string[] | null       // null = свободный ответ

  

channelTypes: ChannelType[]       // перечень доступных каналов в этом шоу

  

privateChannelRules:              // правила инициации приватных каналов

  initiator: "host_only" | "character_request_host_approves" | "character_free"

  maxPrivatesPerPhase: number     // лимит приватных разговоров на всю фазу

  maxPrivatesPerCharacterPerPhase: number  // лимит на одного персонажа в фазе

  requestQueueMode: "fifo" | "host_priority"  // если несколько запросов одновременно

  requestFormat: "public_ask" | "structured_signal"

    // public_ask: персонаж просит публично ("хочу поговорить с X")

    // structured_signal: персонаж возвращает intent: "request_private" в CharacterResponse

scoringRules: ScoringRule[] | null

winCondition: WinConditionExpression | null

```

  

**Acceptance criteria:**

- Движок принимает любой валидный шаблон и запускает выпуск по его правилам

- Смена шаблона не требует изменения кода движка

- MVP поставляется с одним встроенным шаблоном (см. раздел 12)

  

### 4.3 Character Definition (Определение персонажа)

  

Описание роли, независимое от конкретной LLM-модели.

  

**Поля:**

```

id: string

name: string

publicCard: string                    // то, что все знают о персонаже

personalityPrompt: string             // системный prompt: характер, стиль, ценности

motivationPrompt: string              // цели, стимулы, что персонаж хочет выиграть

boundaryRules: string[]               // чего персонаж НЕ делает (жёсткие ограничения)

startingPrivateContext: PrivateContext // приватные вводные, выдаются персонажу на старте

speakFrequency: "low" | "medium" | "high"  // влияет на очерёдность ходов

responseConstraints:

  maxTokens: number

  format: "free" | "structured" | "choice"

  language: string

```

  

**PrivateContext (стартовый):**

```

secrets: string[]                     // факты, известные только этому персонажу

alliances: AllianceRecord[]           // предварительные договорённости (до начала шоу)

goals: string[]                       // приватные цели

wildcards: string[]                   // "козыри" — информация, раскрываемая стратегически

```

  

`startingPrivateContext` — неизменяемый снимок на старте выпуска, хранится в `show_characters.private_context`. Оркестратор и Context Builder используют его как базовый слой памяти персонажа.

  

**Non-MVP расширение: динамические приватные вводные**

Описано в разделе 13. Суть: формат шоу и/или ведущий могут выдавать персонажу новые приватные данные (секреты, козыри, задания, условия) в процессе игры — через события типа `private_injection`, видимые только адресату. Расширение не требует изменения MVP-структуры `PrivateContext`, только добавляет новый `event_type` в журнал и соответствующую логику в Host Module.

  

**Acceptance criteria:**

- Персонаж может быть создан как конфигурационный файл без написания кода

- Один и тот же персонаж запускается с разными ModelAdapter без изменения его определения

- `startingPrivateContext` фиксируется при старте шоу и не изменяется в течение выпуска (MVP)

  

### 4.4 Model Adapter (Адаптер модели)

  

Тонкий слой, связывающий Character с конкретным LLM API. Реализует единый интерфейс независимо от провайдера — движок не знает, с кем говорит.

  

**Контракт адаптера (концептуально):**

- Принимает `PromptPackage` (сформированный Context Builder'ом) и возвращает `CharacterResponse`

- Логирует сырой запрос и ответ в таблицу `llm_calls`

- Оценивает количество токенов до вызова (для бюджетного контроля)

- MVP-реализации: `MockAdapter` (возвращает hardcoded ответы) и `OpenAIAdapter`

  

**PromptPackage** содержит:

- System prompt (personality + motivation + boundaries + инструкция формата ответа)

- Контекст персонажа (его информационная тюрьма, сформированная Context Builder'ом)

- Текущий триггер / запрос оркестратора

- Ограничения ответа (maxTokens, format)

  

TypeScript-интерфейс — в Appendix A.

  

**Acceptance criteria:**

- MVP включает MockAdapter и OpenAIAdapter (gpt-4o, gpt-4o-mini)

- Добавление нового провайдера = реализация интерфейса ModelAdapter без изменения движка

- Адаптер логирует сырой запрос и ответ для дебага и Replay

- Переключение через `ADAPTER_MODE=mock|openai` в `.env`

  

### 4.4а CharacterResponse — внутренний протокол ответа

  

Все LLM-вызовы возвращают структурированный ответ. Это не про названия полей — это про то, что оркестратору нужны машиночитаемые сигналы, иначе управление каналами и решениями превращается в разбор текста.

  

**Поля CharacterResponse:**

```

text: string           // реплика персонажа (то, что "говорит" вслух — видно другим)

intent: string         // сигнал оркестратору, не произносится публично (default: "speak")

target: string?        // characterId — к кому обращён intent (если применимо)

decisionValue: string? // заполняется только в Decision Phase

```

  

**Допустимые значения intent (MVP):**

```

"speak"           — просто говорит (дефолт)

"request_private" — просит приватного разговора с target; оркестратор принимает/отклоняет по privateChannelRules

"reveal_wildcard" — раскрывает козырь из startingPrivateContext.wildcards

"end_turn"        — нечего добавить, пропускает ход

```

  

**Non-MVP intent-запросы (инициатива персонажа):**

```

"request_to_speak"   — персонаж хочет взять слово вне очереди

"request_interrupt"  — персонаж хочет прервать текущего говорящего

```

Оба обрабатываются только если шаблон разрешает `allowCharacterInitiative: true`. Оркестратор принимает/отклоняет по лимитам и правилам фазы — подробнее в разделе 13.

  

**Как используется оркестратором:**

- `intent` и `target` читает оркестратор; другие персонажи их не видят

- `request_private` ставится в очередь по `privateChannelRules`

- `reveal_wildcard` — оркестратор публикует козырь в PUBLIC, помечает как раскрытый

- Невалидный JSON → retry (max 2) → fallback `{ text: "[молчит]", intent: "end_turn" }`; fallback логируется с `fallback: true` в metadata

  

TypeScript-интерфейс — в Appendix A.

  

**Acceptance criteria:**

- Все ответы в Decision Phase содержат `decisionValue`

- Запросы приватных разговоров идут через `intent`, не через разбор текста

- Fallback не крашит оркестратор

- Non-MVP intent'ы (`request_to_speak`, `request_interrupt`) игнорируются оркестратором в MVP, если шаблон не включает `allowCharacterInitiative`

  

### 4.5 Context Builder (Сборщик контекста)

  

Модуль, который формирует контекст персонажа перед каждым LLM-вызовом.

  

**Зачем это нужно явно:** LLM не хранит никакой памяти о мире между вызовами. Движок обязан формировать полный контекст заново на каждый ход — это единственный способ обеспечить приватность (разные персонажи видят разное), контролировать стоимость (не гнать весь журнал в каждый запрос) и дебажить (можно воспроизвести точный контекст любого вызова).

  

**Двухслойная модель памяти:**

  

*Слой 1 — Sliding Window (события):*

Последние `contextWindowSize` событий, видимых персонажу. Фильтрация по `audience_ids`: включаются только события, где персонаж в списке аудитории. Это "оперативная память" — что происходило недавно.

  

*Слой 2 — Facts List (структурная долгосрочная память):*

Явный список фактов, которые персонаж знает и помнит независимо от длины журнала. Формируется и обновляется из журнала событий при каждом ходу. Содержит:

- секреты и козыри из `startingPrivateContext` (и динамические вводные в Non-MVP)

- обещания, данные в приватных разговорах ("договорились с X голосовать за Y")

- публично установленные факты, на которые персонаж явно реагировал

- раскрытые козыри (свои и чужие)

  

**Алгоритм сборки на каждый ход:**

1. Взять `startingPrivateContext` персонажа

2. Обновить Facts List: пройти по новым событиям журнала с момента последнего обновления, извлечь факты (детерминированно — без LLM в MVP)

3. Отфильтровать Sliding Window из журнала (последние `contextWindowSize` видимых событий)

4. Собрать итоговый контекст: Facts List + Sliding Window + текущий триггер

5. Проверить, что итог укладывается в `maxContextTokens`; если нет — сократить Sliding Window (Facts List не обрезается)

  

**MVP-политика:**

- Только Sliding Window + базовый Facts List (ручное извлечение фактов из событий, без LLM-саммаризации)

- `contextWindowSize` — параметр в шаблоне формата (рекомендуемый дефолт: 30 событий)

- LLM-саммаризация — Non-MVP

  

**Acceptance criteria:**

- Персонаж никогда не получает содержание чужих приватных разговоров

- Facts List содержит `startingPrivateContext` и все раскрытые козыри

- Лог Context Builder'а позволяет воспроизвести точный контекст любого вызова по номеру события

  

### 4.6 Event Journal (Журнал событий)

  

Append-only лог всех событий выпуска. Единственный источник истины для Context Builder, Replay, дебага и будущего визуального слоя.

  

**Поля события:**

```

id: string              — uuid

showId: string

timestamp: number       — unix ms

sequenceNumber: number  — порядковый номер в выпуске (монотонный)

phase: string           — id текущей фазы

type: EventType         — см. ниже

channel: string         — 'PUBLIC' | 'PRIVATE' | 'ZONE' (источник истины)

visibility: string      — derived из channel; не редактируется независимо

senderId: string        — characterId или "host"

receiverIds: string[]   — явные адресаты ([] = все в канале)

audienceIds: string[]   — кто фактически видит событие (денормализовано для Context Builder)

content: string         — текст реплики или описание события

metadata: object        — токены, модель, задержка, fallback-флаг и т.д.

seed: string            — seed этого вызова

```

  

**Типы событий (EventType):**

```

"speech"            — реплика персонажа

"host_trigger"      — ведущий выдаёт триггер / вопрос / объявление

"phase_start"

"phase_end"

"channel_change"    — смена канала (отвели поговорить / вернули)

"decision"          — финальное решение персонажа

"revelation"        — раскрытие решения или козыря публично

"private_injection" — [Non-MVP] ведущий выдаёт персонажу новые приватные вводные;

                      видно только адресату (audienceIds = [characterId])

"system"            — техническое событие: пауза, откат, ошибка, fallback, budget_saving_mode, graceful_finish

```

  

**Acceptance criteria:**

- Любой момент выпуска воспроизводится, прочитав журнал от начала до нужного sequenceNumber

- Откат фазы = удаление событий после checkpoint + сброс состояния оркестратора

- Журнал экспортируется в JSON

- Событие `private_injection` структурно поддерживается в схеме, даже если логика выдачи — Non-MVP

  

### 4.7 Host Module (Модуль ведущего)

  

Детерминированный модуль, реализующий логику ведущего по правилам шаблона.

  

**Функции:**

- Инициализация выпуска: загрузка шаблона, персонажей, генерация seed

- Управление очередью ходов (speakFrequency + правила шаблона)

- Выдача триггеров из TriggerTemplate фазы

- "Отвести поговорить": переключение канала на PRIVATE для подмножества персонажей

- Запуск Decision Phase: рассылка запросов на финальное решение по DecisionConfig

- Revelation: раскрытие решений по правилам шаблона

  

**Опциональный LLM-ведущий (пост-MVP):**

Если в конфиге указан `hostVoiceAdapter`, Host Module передаёт ему только текст для озвучки триггеров/объявлений. LLM-ведущий не имеет права изменять правила, считать очки или принимать решения за участников.

## 8. Соображения по безопасности и контролю затрат

  

### Контроль токенов (критично для MVP)

  

**Бюджет и три порога:**

  

```

0–79%   — нормальный режим; оркестратор работает по правилам шаблона без изменений

80–99%  — budget_saving_mode (режим экономии)

100%    — graceful_finish (принудительное завершение)

```

  

**budget_saving_mode (80%):**

Оркестратор не просто логирует предупреждение — он меняет поведение:

- Сокращает `maxTokens` на реплику (до 50% от исходного значения)

- Уменьшает оставшееся число ходов в текущей и последующих фазах

- Запрещает необязательные действия: новые приватные разговоры (сверх уже открытых), раскрытие козырей через triggered-окно

- Повышает приоритет перехода к Decision Phase — если фаза ещё не завершена, оркестратор ускоряет `completionCondition`

- Фиксирует событие `event_type: "system"` с пометкой `budget_saving_mode: true`

  

**graceful_finish (100%):**

Оркестратор не обрывает выпуск аварийно. Алгоритм:

1. Завершает текущую микрозадачу (доводит активный ход до конца)

2. Закрывает открытые приватные каналы (возвращает всех в PUBLIC)

3. Если текущая фаза не Decision — пропускает оставшиеся фазы и немедленно запускает Decision Phase по правилам формата

4. Собирает финальные решения от всех персонажей (без новых реплик)

5. Публикует Revelation и завершает выпуск

6. Фиксирует событие `event_type: "system"` с `graceful_finish: true`

  

**Лимиты на уровне персонажа:**

- `maxTokens` в `responseConstraints` — лимит одной реплики в нормальном режиме

- Рекомендация MVP: 150 токенов на реплику

  

**Оценка стоимости MVP-выпуска (gpt-4o-mini):**

- 5 персонажей × 20 ходов × ~200 токенов контекста + 150 токенов ответа ≈ 35 000 токенов

- Стоимость: ~$0.02–0.05 за выпуск

  

**Acceptance criteria:**

- При достижении 80% бюджета оркестратор входит в `budget_saving_mode`; выпуск продолжается

- В `budget_saving_mode` реплики короче, необязательные приватки не открываются

- При достижении 100% запускается `graceful_finish`; выпуск всегда доходит до финального решения без аварийного обрыва

- Оба порогых события фиксируются в журнале с соответствующими метками

  

### Безопасность ключей

  

- API ключи только через переменные окружения, никогда в коде

- `.env` в `.gitignore`

- Логи LLM-вызовов не выводятся в консоль по умолчанию (только в БД)

  

### Воспроизводимость: Replay vs Rerun

  

Система поддерживает два режима воспроизведения. Это разные вещи — важно не смешивать.

  

**Replay (гарантированный):**

- Проигрываем сохранённые raw-ответы из таблицы `llm_calls`

- Новых LLM-вызовов нет — результат идентичен оригиналу побайтово

- Это единственный способ показать зрителям "тот же выпуск"

- Используется для: визуализации, озвучки, Camera Director, дебага

  

**Rerun (псевдодетерминизм):**

- Запускаем новый выпуск с тем же seed + шаблон + персонажи

- seed передаётся в LLM API (если провайдер поддерживает), но гарантий нет

- Результат будет похожим, но не идентичным — это нормально

- Используется для: экспериментов, сравнения моделей, "а что если"

  

**Rollback в DEBUG-режиме:**

- Rollback гарантирован только в Replay-режиме: откатываем cursor в `llm_calls`, переигрываем из сохранённых ответов

- В Rerun-режиме rollback = новая ветка выпуска (другие LLM-ответы после точки отката), это задокументированное поведение, не баг

- В таблице `shows` хранится `seed` для Rerun и флаг `replayAvailable: boolean`

## 10. Потенциальные проблемы и решения

  

### P1: LLM "выходит из роли"

  

**Проблема:** персонаж игнорирует boundaryRules, отвечает не в характере или возвращает невалидный JSON.

  

**Решение:**

- Добавить в systemPrompt explicit reminder ограничений + инструкцию формата ответа (CharacterResponse) перед каждым вызовом

- Post-processing: парсить CharacterResponse, валидировать обязательные поля

- Если JSON не парсится — retry (max 2), затем fallback `{ text: "[молчит]", intent: "end_turn" }`

- Fallback логируется как `event_type: "system"` с пометкой `fallback: true` в metadata

  

### P2: Контекст растёт и превышает лимит токенов

  

**Проблема:** длинные шоу → большой журнал → дорого и медленно.

  

**MVP-политика (только Sliding Window + Facts List):**

- Context Builder использует двухслойную модель (см. раздел 4.5): Facts List не обрезается, Sliding Window сокращается при переполнении

- `contextWindowSize` — параметр в шаблоне (дефолт: 30 видимых событий)

- LLM-саммаризация в MVP **не используется**: это лишний вызов, который усложняет дебаг и повышает стоимость

  

**Non-MVP расширение:**

- LLM-генерируемый summary при `phase_end`, хранится как `event_type: "phase_summary"` в журнале

- Context Builder подставляет summary вместо событий старше окна

  

### P3: Rollback в DEBUG-режиме меняет результат

  

**Проблема:** после rollback и перегенерации LLM вернёт другой ответ — выпуск пойдёт по другой ветке.

  

**Решение:**

- Это задокументированное поведение, не баг: в Rerun-режиме rollback = новая ветка

- В Replay-режиме rollback гарантирован: используем сохранённые raw-ответы из `llm_calls`, новых вызовов нет

- В дебаг-интерфейсе режим (Replay/Rerun) отображается явно, чтобы оператор понимал последствия отката

  

### P4: Перерасход бюджета при параллельных вызовах

  

**Проблема:** если несколько персонажей вызываются одновременно, бюджет может превыситься.

  

**Решение (MVP):** Последовательные вызовы — один персонаж за раз. Параллелизм — пост-MVP.

## 12. Шаблоны форматов шоу для MVP

  

### Формат A: "Коалиция" (рекомендуется для MVP)

  

**Концепция:** 5 персонажей соревнуются за один приз. До начала у некоторых есть тайные предварительные договорённости. В финале каждый тайно голосует за другого участника, кто получит приз.

  

**Почему подходит для MVP:**

- Простая структура (3 фазы)

- Финальное решение чёткое (тайное голосование)

- Естественно порождает интригу через приватные разговоры

- Легко верифицировать правильность работы системы

  

**Механический твист (обязательный, не сценарный):**

Один из 5 персонажей получает в `privateContext.wildcards` козырь — факт, способный изменить чужие решения, если раскрыть его публично. Козырь можно раскрыть только в конце Фазы 2 (оркестратор открывает окно через триггер). Раскрытие происходит через `intent: "reveal_wildcard"` в CharacterResponse — оркестратор публикует козырь в PUBLIC и помечает его как раскрытый. Это не сценарий — персонаж сам решает, раскрывать или нет.

  

**Фазы:**

```

Фаза 1: Знакомство (PUBLIC, 3 хода на персонажа, sequential)

  — Каждый представляется публично

  — Триггер: "Расскажи о себе и своих ценностях"

  — Каналы: только PUBLIC

  — Приватные разговоры: запрещены

  

Фаза 2: Переговоры (PUBLIC + PRIVATE, 15 ходов суммарно, frequency_weighted)

  — Персонажи общаются публично и могут запрашивать приватные разговоры

  — Запрос приватного разговора: через intent: "request_private" + target

  — Ведущий одобряет запрос (host_approves), пара уходит в PRIVATE на 3 хода

  — Лимиты приваток: maxPrivatesPerPhase: 4, maxPrivatesPerCharacterPerPhase: 2

  — В конце фазы (последние 2 хода): ведущий выдаёт триггер "Время раскрыть козыри"

    Носитель козыря решает: reveal_wildcard или промолчать

  

Фаза 3: Финальное решение (DECISION)

  — Тайное, одновременное голосование

  — Формат: choice из [имена других персонажей]

  — Раскрытие: после всех ответов, публично

```

  

**privateChannelRules:**

```json

{

  "initiator": "character_request_host_approves",

  "maxPrivatesPerPhase": 4,

  "maxPrivatesPerCharacterPerPhase": 2,

  "requestQueueMode": "fifo",

  "requestFormat": "structured_signal"

}

```

  

**DecisionConfig:**

```json

{

  "timing": "simultaneous",

  "visibility": "secret_until_reveal",

  "revealMoment": "after_all",

  "format": "choice"

}

```

### Формат B: "Дилемма" (альтернатива)

  

**Концепция:** 5 персонажей попали в общую ситуацию, где есть коллективное решение с индивидуальными последствиями. Один из участников знает "секрет", который меняет расклад, если раскроет его.

  

**Почему интересен:**

- Тестирует механику раскрытия секрета как стратегического действия

- Дилемма заключённого на уровне группы

- Финальное решение — не голосование, а структурированный выбор (сотрудничать / предать)

  

**Фазы:**

```

Фаза 1: Ознакомление с условиями (PUBLIC, 3 хода)

  — Ведущий объявляет условия и ставки

  

Фаза 2: Совещание (PUBLIC + PRIVATE, 8 ходов)

  — Персонажи обсуждают стратегию

  — Носитель секрета решает: раскрыть или нет

  

Фаза 3: Индивидуальный выбор (DECISION)

  — Последовательное, публичное

  — Формат: choice из ["сотрудничать", "предать"]

  — Каждый видит предыдущие выборы перед своим

```

  

**Примечание:** Формат B лучше демонстрирует каскадную динамику (каждый видит предыдущих), но сложнее балансировать — рекомендован как второй шаблон после успешного запуска Формата A.

## 13. Out of Scope / Non-MVP

  

Этот раздел фиксирует функции, которые **не реализуются в MVP**, но архитектурно предусмотрены. Упоминание здесь означает: движок не должен их блокировать, но и не должен их реализовывать.

  

### 13.1 Инициатива персонажа

  

**Суть:** персонаж может сигнализировать желание взять слово или прервать другого — через `intent` в CharacterResponse. Оркестратор решает, удовлетворить запрос или нет.

  

**Почему не MVP:** усложняет управление темпом и создаёт риск бесконечных запросов на слово. В MVP оркестратор всегда назначает ходы сам.

  

**Как работает (Non-MVP):**

- Шаблон включает флаг `allowCharacterInitiative: true` и параметры лимитов:

  - `maxInitiativeRequestsPerPhase: number` — сколько раз персонаж может запросить слово за фазу

  - `initiativeGrantMode: "fifo" | "host_priority" | "random"` — порядок удовлетворения запросов

- Персонаж возвращает `intent: "request_to_speak"` или `intent: "request_interrupt"` в своём ответе на назначенный ход

- Оркестратор ставит запрос в очередь и обрабатывает по `initiativeGrantMode` в ближайший подходящий момент (не нарушая текущий ход)

- `request_interrupt` — более приоритетный запрос; оркестратор может завершить текущий ход досрочно, если шаблон это разрешает (`allowInterrupt: true`)

- Лимиты соблюдаются; сверхлимитные запросы отклоняются без уведомления персонажа

  

**Acceptance criteria (Non-MVP):**

- При `allowCharacterInitiative: false` (MVP-дефолт) оркестратор игнорирует `request_to_speak` и `request_interrupt` без ошибок

- При `allowCharacterInitiative: true` запросы обрабатываются по лимитам; темп шоу не ломается

- Все запросы и решения по ним логируются в журнал

  

### 13.2 Динамические приватные вводные

  

**Суть:** формат шоу и/или ведущий могут выдавать персонажу новые секреты, козыри, задания или условия в процессе игры — не только на старте.

  

**Почему не MVP:** стартового `startingPrivateContext` достаточно для первого формата. Динамические вводные усложняют Context Builder и требуют дополнительной логики в Host Module.

  

**Как работает (Non-MVP):**

- Шаблон описывает `injectionTriggers` — условия, при которых ведущий выдаёт приватные вводные:

  ```

  injectionTriggers:

    - triggerCondition: ConditionExpression  // например: "phase_end AND phase.id == 'negotiations'"

      targetCharacterId: string | "random" | "wildcard_holder"

      injectionType: "secret" | "wildcard" | "alliance" | "private_task" | "condition_change"

      content: string                        // текст вводной

  ```

- Ведущий создаёт событие `event_type: "private_injection"` в журнале

- `audienceIds` события = только `[targetCharacterId]` — другие персонажи не видят ни факта выдачи, ни содержания

- Context Builder на следующем ходу адресата включает новые вводные в Facts List

- Структура `show_events` уже поддерживает этот event_type (MVP)

  

**Acceptance criteria (Non-MVP):**

- `private_injection` события логируются корректно и видны только адресату

- Context Builder обновляет Facts List адресата после получения вводной

- Журнал содержит полную историю выданных вводных (для аудита ведущего/оператора)

  

### 13.3 LLM-саммаризация контекста

  

При переходе фазы оркестратор может вызвать LLM для генерации краткого summary событий фазы. Summary хранится как `event_type: "phase_summary"` и используется Context Builder'ом вместо событий старше окна.

  

### 13.4 Параллельные LLM-вызовы

  

В MVP вызовы последовательные (один персонаж за раз). Non-MVP: несколько персонажей вызываются параллельно в рамках одного хода (например, в Decision Phase все пишут одновременно).

  

### 13.5 Мульти-провайдер в одном шоу

  

Разные персонажи используют разных LLM-провайдеров. Например: Персонаж A — GPT-4o, Персонаж B — Claude.

  

### 13.6 LLM-ведущий (голос и стиль)

  

Опциональный `hostVoiceAdapter`: LLM генерирует текст подводок и объявлений ведущего. Не имеет права менять правила, считать очки или принимать решения. Только озвучка детерминированных решений оркестратора.
